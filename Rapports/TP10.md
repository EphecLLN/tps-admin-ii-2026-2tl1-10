# Partie 1 Mise en place de l'environnement

**Questions** : 

*1. Quelles sont les endpoints/routes disponibles au niveau des 2 applications ?*

En regardant le fichier `app.py`, on voit que 3 routes (endpoints) ont été créés avec Flask : 
- `/` **(La racine)** : Renvoie un simple message d'accueil et génère un log de      niveau `INFO`
- `/action` : Simule le déclenchement d'une action sensible et génère un log de niveau `WARNING`.
- `/error` : Elle génère de manière intentionnel une erreur de division par zéro (`10/0`) pour tester la capture des exceptions, ce qui produit un log de niveau `ERROR` avec la trace complète de l'erreur (traceback). 
	
*2. Les 2 applications ne sont pas exactement similaires. Quelles sont les différences entre les 2 ?*

La différences entre ces 2 applications se joue au niveau de la configuration des variables d'environnement dans le fichier `docker-compose.yml` : 
- **Les ports exposés** : `flask_app_1` est accessible sur le port `5001` de l'hôte, tandis que `flask_app_2` est sur le port `5002`
- **L'identifiant (`APP_INSTANCE_ID`)** : L'une est nommé `app1` et l'autre `app2` dans les logs
- **Le décalage temporel (`APP_TIMESTAMP_OFFSET_SECONDS`)** : C'est la différence la plus notable entre les 2 applications, `app1` n'a aucun décalage (0), mais `app2` a un décalage de `-3600` secondes (soit 1 heure dans le passé).

*3. Quels sont les logs générés par les applications Flask ? Quel est leur format ?*

- **Contenu des logs** : Les applications génèrent un log au démarrage, puis des logs spécifiques pour chaque route visitée. Les logs incluent des champs classiques (niveau de log, message) mais aussi des champs personnalisés (injectés via l'attribut `extra` en Python) comme `source_app`, `event_timestamp`, `client_ip`, ou encore des contextes d'erreur (`error_code`).
- **Le format** : Grâce à la librairie python-logstash-async et à la configuration de Logstash, les logs sont structurés et envoyés au format **JSON**. Le timestamp de l'événement est formaté en norme ISO8601 UTC.

*4. Quel est le flux de données et l'architecture de la stack ?*

**Le flux de données :**

1. Les conteneurs **Flask** génèrent des logs et les envoient de manière asynchrone via le réseau (TCP).
2. **Logstash** écoute sur le port 5000, réceptionne ces lignes JSON, les traite (pipeline d'ingestion), et les pousse vers Elasticsearch.
3. **Elasticsearch** (sur le port 9200) reçoit ces données de Logstash, les indexe et les sauvegarde.
4. L'utilisateur se connecte à l'interface web de **Kibana** (port 5601), qui va requêter Elasticsearch pour lire et afficher les données sous forme de tableaux de bord.

**Le stockage physique :**
Les logs sont stockés physiquement sur le serveur hôte. Dans le `docker-compose.yml`, on voit un volume nommé `es_data` qui est monté sur `/usr/share/elasticsearch/data`. Docker gère ce volume et écrit les données de la base Elasticsearch directement sur le disque de la machine hôte pour qu'elles ne soient pas perdues si le conteneur redémarre.

**Le rôle de chacun :**
- **Logstash** : Il réceptionne les paquets (logs), s'assure qu'ils sont au bon format (JSON), et les achemine à la bonne destination.
- **Elasticsearch** : Il stocke les logs dans des tiroirs classés par jour (l'index `flask-logs-YYYY.MM.dd`) et permet de chercher à l'intérieur avec une très grande vitesse.
- **Kibana** : C'est l'interface graphique qui permet de créer de beaux graphiques et de lire ce qui est stocké dans la bibliothèque Elasticsearch sans avoir à taper des requêtes complexes dans un terminal.
	
# Partie 2 : Génération et visualisations des logs 

**Exploration de Kibana**

*1. Valeurs des champs pour la route `/action` de `flask_app_1` :*
En regardant la fonction `@app.route('/action')` dans le code et les variables d'environnement du conteneur, on observe :

- `level` : `WARNING` (défini par le `logging.WARNING` dans le code).
- `message` : `"Une action sensible a été déclenchée."`
- `extra.source_app` : `"app1"` (récupéré via la variable d'environnement APP_INSTANCE_ID de flask_app_1).
- `extra.user_id` : `"dev_user"` (injecté manuellement via le dictionnaire `extra_fields`).

*2. Deux autres champs présents dans ce log :*
En étendant le document JSON dans Kibana, on peut voir de nombreux champs générés par la librairie Logstash. Parmi eux ont peut y trouver :
- `extra.action_name` : `"trigger_event"` (ajouté en même temps que le user_id dans le code).
- `@timestamp` et `extra.event_timestamp` : Pour l'application 1, ces deux valeurs seront presque identiques (à 1 ou 2 secondes près, à cause du `time.sleep`générant une latence aléatoire). Par exemple :
    - `@timestamp` : `2026-04-22T18:30:05.000Z` (Heure de réception par Logstash).
    - `extra.event_timestamp` : `2026-04-22T18:30:03.000Z` (Heure de création par Flask).

**Timestamps et synchronisation des horloges**

*3. Différence majeure entre les timestamps pour `flask_app_2` :*
Pour les logs générés par flask_app_2, on observe un **décalage d'environ 1 heure (3600 secondes)** entre les deux.
- `extra.event_timestamp` affichera une heure dans le passé (car la variable `APP_TIMESTAMP_OFFSET_SECONDS` est à `-3600`).
- `@timestamp` affichera l'heure actuelle et réelle (car Logstash lui, a la bonne heure).

*4. Lequel de ces timestamps est le plus trompeur si utilisé seul (sans synchronisation) ?*
C'est `extra.event_timestamp` qui est le plus trompeur dans ce cas précis.
- C'est lui parce que si on tris tous tes logs uniquement sur ce champ, un événement qui s'est produit sur `app2` après un événement sur `app1` apparaîtra chronologiquement 1 heure avant dans Kibana. Cela détruit totalement la causalité des événements (impossible de savoir si l'erreur de l'app 2 a causé le crash de l'app 1, ou inversement).

*5. Pourquoi NTP est crucial et quel timestamp devient le plus fiable ?*
- **Pourquoi NTP est crucial :** Dans une architecture distribuée (microservices, clusters), les requêtes rebondissent de serveur en serveur. Si leurs horloges ne sont pas parfaitement synchronisées (via le protocole NTP), il est impossible de corréler les logs entre eux pour retracer le parcours d'une requête ou diagnostiquer l'origine d'une panne globale.
- **Quel timestamp est le plus fiable (avec NTP)** : Si toutes les machines sont parfaitement synchronisées, alors `extra.event_timestamp` devient le meilleur timestamp. Pourquoi ? Parce qu'il représente le moment exact où l'événement a eu lieu au cœur de l'application. Le `@timestamp` de Logstash, lui, inclut le temps de trajet sur le réseau (latence) et le temps d'attente dans la file de traitement de Logstash, il est donc légèrement en retard par rapport à la réalité.

# Partie 3 : Filtrage et recherche dans Kibana

**Filtrage et recherche dans Kibana**

*1. Quelle requête KQL utiliseriez-vous pour afficher uniquement les messages de niveau WARNING de flask_app_2 qui contiennent la chaîne de caractères "sensible" dans le champ message ?*

`level : "WARNING" and extra.source_app : "app2" and message : sensible`

*2. Principal avantage d'Elasticsearch par rapport à des fichiers texte sur chaque serveur ?*

Le principal avantage est la **centralisation et l'indexation structurée.**
Si on utilisait des fichiers texte, on aurait dû se connecter en SSH sur chaque serveur, utiliser des commandes comme `grep` pour chercher péniblement dans des gigaoctets de texte brut, et essayer de fusionner mentalement les résultats de plusieurs machines.
Avec Elasticsearch, tous les logs de toutes les applications convergent au même endroit. Comme les données sont structurées en JSON (chaque info est dans une "case" spécifique) et indexées (comme le sommaire d'un livre), une recherche sur plusieurs mois ou sur 50 serveurs différents te donne un résultat corrélé en quelques millisecondes, via une interface graphique unique (Kibana).

**Synchronisation des horloges et NTP**

*3. Dans le cas de containers Docker qui partagent le noyau du même hôte, qu’en est-il de cette problématique de synchronisation ?*

Les conteneurs ne sont pas des machines indépendantes à ce niveau-là : ils partagent le noyau (kernel) de la machine hôte. Par conséquent, **ils utilisent nativement et obligatoirement l'horloge de l'hôte**. Il n'y a donc aucun risque de désynchronisation entre les conteneurs d'un même serveur. Si l'hôte est à l'heure, tous ses conteneurs le sont aussi. Il ne faut jamais installer de client NTP à l'intérieur d'un conteneur.

*4. Même question dans le cas de containers appartenant à une stack Docker Swarm tournant sur plusieurs nœuds ?*

Ici, les conteneurs sont répartis sur des machines physiques (ou virtuelles) différentes. Chaque nœud (serveur) a son propre noyau et sa propre horloge matérielle qui va dériver à son propre rythme. Il est donc **strictement impératif de synchroniser chaque nœud du Swarm via NTP**. Si on ne le fait pas, les logs des conteneurs du Nœud A et du Nœud B n'auront pas de cohérence chronologique, et les mécanismes internes de Swarm (élection de leaders, routage réseau) peuvent même planter.

*5. Qu’en est-il de machines virtuelles et de leur hyperviseur ?*

Les horloges des machines virtuelles (VM) sont émulées en logiciel. À cause du partage des ressources CPU par l'hyperviseur, **les horloges des VM ont tendance à dériver beaucoup plus vite et de manière plus erratique** qu'une vraie machine physique. Pour gérer ça, on utilise généralement deux méthodes combinées : les "Guest Tools" (comme VMware Tools ou qemu-guest-agent) qui resynchronisent l'horloge de la VM sur celle de l'hyperviseur, couplés à un démon NTP standard installé dans l'OS de la VM pour une précision fine.

*6. Comment assurer la synchronisation dans l’infrastructure des TPs 4 à 7 (DNS public, Web, Mail) ?*

Pour ces services d'infrastructure classiques et souvent isolés sur internet, la méthode standard suffit : on installe et active un client NTP léger (comme `chrony` ou `systemd-timesyncd` sous Linux) sur chaque VPS. On les configure pour pointer vers des serveurs de temps publics fiables (comme `pool.ntp.org`). C'est vital ici car les certificats TLS (Web/Mail), la validité des signatures DNSSEC, et les filtres antispam s'appuient massivement sur des timestamps précis pour valider la sécurité.

*7. Comment assurer la synchronisation dans l’infrastructure des TPs 8 et 9 (Haute disponibilité, Microservices, Message Broker) ?*

On entre dans des architectures critiques et très distribuées. Les bases de données en cluster ou les message brokers (comme RabbitMQ/Kafka) détestent la latence et les sauts temporels : si un nœud a une milliseconde de retard, un cluster peut se diviser (split-brain) ou corrompre l'ordre des messages.
Ici, pointer vers internet ne suffit plus. Il faut déployer **un serveur NTP local (Stratum 2 ou 3)** dédié à l'intérieur de l'infrastructure privée. Ainsi, tous les microservices et nœuds du cluster se synchronisent sur ce serveur local spécifique, garantissant que toutes les machines partagent exactement la même microseconde, sans dépendre des aléas de latence du réseau public internet.
